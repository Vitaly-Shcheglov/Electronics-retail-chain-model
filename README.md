# Проект "Онлайн платформа торговой сети электроники"/"Electronics retail chain model"

## Описание проекта

Electronics retail chain model — веб-приложение для управления онлайн платформой торговой сети электроники. Система включает в себя API-интерфейс, админ-панель и базу данных для управления иерархической структурой сети, состоящей из заводов, розничных сетей и индивидуальных предпринимателей.

## Структура проекта

- Модели:
  - NetworkNode: Представляет узлы сети (заводы, розничные сети, индивидуальные предприниматели).
  - Supplier: Представляет поставщиков, которые снабжают узлы сети.
  - NetworkObject: Представляет объекты сети, которые связаны с узлами.
  - Product: Модель для продуктов, предлагаемых в сети.

## Требования к реализации

### 1. Иерархическая структура сети

Сеть должна представлять собой иерархическую структуру из трех уровней:

- Завод (уровень 0)
- Розничная сеть (уровень 1)
- Индивидуальный предприниматель (уровень 2)

Каждое звено сети ссылается на одного поставщика (не обязательно предыдущего по иерархии). Уровень объекта определяется его позицией в цепочке поставок.

### 2. Элементы узлов сети

Каждое звено сети должно обладать следующими элементами:
- Название
- Контакты: email, страна, город, улица, номер дома
- Продукты: название, модель, дата выхода продукта на рынок
- Поставщик (предыдущий по иерархии объект сети)
- Задолженность перед поставщиком с точностью до копеек
- Время создания (автоматически заполняется при создании)

### 3. Админ-панель

- Вывод всех созданных объектов в админ-панели.
- Ссылка на «Поставщика» на странице объекта сети.
- Фильтр по названию города.
- Admin action, очищающий задолженность перед поставщиком у выбранных объектов.

### 4. CRUD операции для модели поставщика

- Запрет на обновление через API поля «Задолженность перед поставщиком».
- Возможность фильтрации объектов по определенной стране.

### 5. Права доступа к API

- Настройка прав доступа так, чтобы только активные сотрудники имели доступ к API.

## Docker-ориентированная разработка
             
1. Сборка образа:
- docker-compose build
Запуск контейнеров:
- docker-compose up -d
Миграции в контейнере Django:
- docker-compose exec web python manage.py migrate
Создание суперпользователя в контейнере:
- docker-compose exec web python manage.py createsuperuser
Сборка статических файлов в контейнере:
- docker-compose exec web python manage.py collectstatic --noinput
Остановка и удаление контейнеров:
- docker-compose down

2. Архитектура и конфигурация Docker (пример)
Файлы:
docker-compose.yml:
сервисы: django (web), db (PostgreSQL), nginx (reverse proxy), celery (если есть задача фоновая1. ).
Dockerfile для Django:
-FROM python:3.x
-WORKDIR /app
-COPY requirements.txt .
-RUN pip install -r requirements.txt
-COPY . .
-CMD ["gunicorn", "project.wsgi:application", "--bind", "0.0.0.0:8000"]
Миграции и статику чаще выполняют в процессе сборки/рантайма внутри контейнера.

3. Обеспечение безопасности и конфигураций
Использовать переменные окружения для секретов и настроек.
В продакшн включить DEBUG=False, ALLOWED_HOSTS настроить.
Настроить CSRF, CORS по мере необходимости.
Настроить бэкенд-логирование и ротацию логов.

4. Мониторинг и resilience
Настроить health check для сервиса Django:
В docker-compose добавьте healthcheck:
test: ["CMD", "curl", "-f", "http://localhost:8000/health/"]
interval: 30s
timeout: 10s
retries: 3
Включить логирование и сбор метрик (Prometheus + Grafana при необходимости).

5. Миграции и фиксы в проде
Применение миграций без простоя:
- docker-compose exec web python manage.py migrate --noinput
Применение фиксов и миграций во время CI/CD:
В пайплайне запускать миграции после деплоя.

6. Бэкап и восстановление БД
Резервное копирование PostgreSQL (локально):
docker exec -t db_pg_container pg_dump -c -U postgres yourdb > backup.sql
Восстановление:
- psql -U postgres -d yourdb -f backup.sql
Типовые команды для удобства

Очистка старых миграций/кешей:

- DELETE /api/network-nodes/{id}/ - Удалить узел сети.

- GET /api/suppliers/ - Получить список поставщиков.

- GET /api/suppliers/{id}/ - Получить информацию о конкретном поставщике.

### Примеры запросов к API для управления сетями

#### 1. Получение списка сетей

**URL:** `/api/networks/`  
**Метод:** GET  

**Шаги в Postman:**
1. Откройте Postman.
2. Выберите метод `GET`.
3. Введите URL: `http://127.0.0.1:8000/api/networks/`.
4. Перейдите на вкладку `Headers` и добавьте заголовок:
   - `Key`: `Authorization`
   - `Value`: `Bearer your_access_token`
5. Нажмите кнопку `Send`.
**Ожидаемый результат:** Вы получите ответ со статусом 200 и список всех сетей в формате JSON.

#### 2. Создание новой сети

**URL:** `/api/networks/`  
**Метод:** POST  

**Шаги в Postman:**
1. Выберите метод `POST`.
2. Введите URL: `http://127.0.0.1:8000/api/networks/`.
3. Перейдите на вкладку `Headers` и добавьте заголовки:
   - `Key`: `Content-Type`
   - `Value`: `application/json`
   - `Key`: `Authorization`
   - `Value`: `Bearer your_access_token`
4. Перейдите на вкладку `Body`, выберите `raw` и установите формат на `JSON`.
5. Вставьте следующий JSON в тело запроса:
   ```json
   {
       "name": "Сеть A",
       "description": "Описание сети A"
   }
   ```
6. Нажмите кнопку `Send`.
**Ожидаемый результат:** Вы получите ответ со статусом 201 и сообщение о том, что сеть была успешно создана.
